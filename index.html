进程-线程-协程

发表于 2020-11-19 更新于 2020-11-21 分类于 基础知识

进程

程序并不能单独运行，只有将程序装载到内存中，系统为它分配资源才能运行，而这种执行的程序称之为进程。程序和进程的区别就在于：程序是指令的集合，它是进程运行的静态描述文本；进程是程序的一次执行活动，属于动态概念。在多道编程中，我们允许多个程序同时加载到内存中，在操作系统的调度下，可以实现并发地执行，大大提高了CPU的利用率。

进程是操作系统分配资源的基本单位。

优点：最大的优点就是稳定性高，因为一个子进程崩溃了，不会影响主进程和其他子进程。

缺点：创建进程的代价大，在Unix/Linux系统下，用fork调用还行，在Windows下创建进程开销巨大。操作系统能同时运行的进程数也是有限的，在内存和CPU的限制下，如果有几千个进程同时运行，操作系统连调度都会成问题。

线程

线程是操作系统能够进行运算调度的最小单位。它被包含在进程之中，是进程中的实际运作单位。一条线程指的是进程中一个单一顺序的控制流，一个进程中可以并发多个线程，每条线程并行执行不同的任务。一个线程是一个execution context（执行上下文），即一个CPU执行时所需要的一串指令。

优点：

  • 能适当提高程序的执行效率

  • 能适当提高资源的利用率（CPU&内存）

  • 线程上得任务执行完后自动销毁

缺点：

  • 开启线程需要占用一定的内存空间(default,512KB), 如开启大量的线程, 会占用大量的内存空间, 降低程序性能

  • 调度时, 要保存线程状态，频繁调度, 需要占用大量的机时, 线程越多，CPU在调用线程上的开销就越大

  • 程序设计更加复杂，比如线程间的通信、多线程的数据共享.

协程

协程，又称微线程。协程是一种用户态的轻量级线程。

协程拥有自己的寄存器上下文和栈。协程调度切换时，将寄存器上下文和栈保存到其他地方，在切回来的时候，恢复先前保存的寄存器上下文和栈。

所以子程序调用是通过栈实现的，一个线程就是执行一个子程序。

子程序调用总是一个入口，一次返回，调用顺序是明确的。而协程的调用和子程序不同。

协程看上去也是子程序，但执行过程中，在子程序内部可中断，然后转而执行别的子程序，在适当的时候再返回来接着执行。

优点：
因为子程序切换不是线程切换，而是由程序自身控制，因此，没有线程切换的开销，和多线程比，线程数量越多，协程的性能优势就越明显。

不需要多线程的锁机制，因为只有一个线程，也不存在同时写变量冲突，在协程中控制共享资源不加锁，只需要判断状态就好了，所以执行效率比多线程高很多。

进程、线程、协程的关系和区别

  • 进程拥有自己独立的堆和栈，既不共享堆，亦不共享栈，进程由操作系统调度。
  • 线程拥有自己独立的栈和共享的堆，共享堆，不共享栈，线程亦由操作系统调度(标准线程是的)。
  • 协程和线程一样共享堆，不共享栈，协程由程序员在协程的代码里显示调度。

任务类型： 计算密集型任务（C语言） 、IO密集型任务（脚本语言）

Serverless应用场景

发表于 2020-11-15 分类于 serverless

Serverless服务的使用场景需要满足两个条件：

  • 应用负载有显著的波峰波谷
  • 基于事件触发的数据处理

下面用几个场景说明serverless服务的使用案例：

一、社交/电商类应用中用函数服务压缩图片

用户通过客户端或移动设备上传图片，图片进行（1）不同分辨率的压缩已适应不同设备（2）需要增加水印 （3）生成缩略图等。通过事件触发处理函数执行，无需为函数保留资源。

主要处理步骤为：

  1. 将图片上传到指定的对象存储桶中
  2. 将用户上传的每个图像的尺寸进行压缩或者图片打水印
  3. 将处理完后的图像上传到另一个指定的对象存储桶中

二、简易中文信息过滤

用分布式消息服务来存储一些重要信息。需要对消息进行处理，判断消息是否适合发布（有些消息不适合发布，例如政治敏感、广告等）。业务爆发时可以自动调度资源运行更多函数实例以满足处理需求。

主要处理步骤为：

  1. 使用DMS服务，在其中创建消息队列，同时在队列中创建消费组，并生产消息
  2. 在函数服务中，创建函数，为函数创建DMS触发器，通过轮询的方式，获取DMS队列中的信息
  3. 在函数中，通过对读取信息进行处理，判断信息是否敏感，是否符合发布
  4. 将过滤后的文字信息通过消息推送服务进行分发

三、由事件触发的批量数据处理

间隔执行或定时触发的ETL（提取->转换->加载）任务。or 机器学习训练数据集预处理，例如游戏AI训练。只有对函数处理文件数据的时间进行计费，无需购买冗余的资源用于非峰值处理。

主要处理步骤为：

  1. 数据汇集到指定的对象存储桶中
  2. 将处理过程的mapreduce按照角色拆分成多个函数
  3. 处理完后的结果存储至缓存数据库供聚合
  4. 使用函数服务聚合结果后传到另一个指定的对象存储桶中

四、经典三层web应用后端

使用函数服务快速构建高可用，自动伸缩的Web/移动应用后端。替代原有虚拟机实例自维护web容器的架构。业务爆发时可以自动调度资源运行更多函数实例以满足处理需求。无请求处理时只需支付少量资源存储费用。利用CDN，NoSQL的高可用性实现网站数据的高可靠性，利用API Gateway和函数服务的高可用性实现网站逻辑的高可用。

https基础原理

发表于 2020-11-08 更新于 2020-11-15 分类于 基础知识

1. 为什么要有https

 在http协议中可能存在信息窃听和身份伪装等安全问题。具体说有以下3点：

  • 通信使用明文（不加密），内容可能会被窃听

  • 不验证通信方的身份，因此有可能遭遇伪装

  • 无法证明报文的完整性，有可能已遭篡改

2. https是什么

        SSL(Secure Socket Layer, 安全套接字层)，及其继任者TLS（Transport Layer Security，传输层安全）是为网络通信提供安全及数据完整性的一种安全协议。如下图所示，SSL位于传输层和应用层之间。

SSL/TLS协议的基本思路：client向server索要publish key，client拿到publish key后用它对数据进行加密后传输至server，最后server再用自己的private key进行解密。存在2个问题：

（1）怎么保证publish key不被篡改；

将publish key 放入证书中，只要保证证书可信即publish key 可信。

（2）公钥加密计算量大，耗时；

client 和 server 都生成一个“会话秘钥”（session key)，用它进行加密。由于“会话秘钥”是对称加密，速度非常快，而服务器的publish key 用来加密session key本身，这样就减少加密运算的时间。

SSL/TLS的最终流程：

（1） 客户端向服务器端索要并验证公钥。

（2） 双方协商生成”对话密钥”。

（3） 双方采用”对话密钥”进行加密通信。

TLS 工作流程如下图所示：

  1. 客户端向服务器发送加密通信请求“client hello”, 同时包含：协议版本，支持加密算法，支持的压缩方法；
  2. 服务端确认客户端请求内容（协议版本，加密算法，压缩算法），同时发送“server hello”, 同时包含： 服务端证书；
  3. 客户端验证服务端证书，客户端握手结束通知，同时向服务端发送客户端证书；
  4. 服务端验证客户端证书，服务端握手结束通知。

3. https怎么工作的

1）客户端发起一个http请求，连接到服务器的443端口。

2）服务端把自己的信息以数字证书的形式返回给客户端（证书内容有公钥，网站地址，证书颁发机构，失效日期等）。证书中有一个公钥来加密信息，而其对应的私钥由服务器持有。

3）验证证书的合法性：客户端收到服务器的响应后先验证证书的合法性（证书中包含的地址与正在访问的地址是否一致，证书是否过期）。

4）生成随机密码（RSA签名）

如果验证通过，或用户接受了不受信任的证书，浏览器就会生成一个随机的对称密钥（session key）并用公钥加密，让服务端用私钥解密，解密后就用这个对称密钥进行传输了，并且能够说明服务端确实是私钥的持有者。

5）生成对称加密算法

验证完服务端身份后，客户端生成一个对称加密的算法和对应密钥，以公钥加密之后发送给服务端。此时被黑客截获也没用，因为只有服务端的私钥才可以对其进行解密。之后客户端与服务端可以用这个对称加密算法来加密和解密通信内容了。

​      总的来说，https的通信步骤就是先通过4次握手验证服务器和客户端的身份同时获取到会话秘钥，然后用对称秘钥进行加密后进行数据的传输。

4. 总结

     https 比http要慢2~100倍，主要是两方面（1）通信慢 （2）大量消耗内存 cpu等资源导致处理速度慢。
